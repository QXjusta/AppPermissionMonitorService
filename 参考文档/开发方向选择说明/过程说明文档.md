项目过程说明

二、当前项目目标
本项目对应“自定义应用权限监控服务（App Permission Monitor Service）”实训课题，目标是在 Android Framework 层实现一个系统级权限监控服务，具备以下能力：
1. 在 SystemServer 中启动服务
2. 通过 Binder/AIDL 向客户端提供查询与回调接口
3. 记录敏感权限使用情况
4. 支持权限校验、历史记录查询与运行时调试

三、当前采用的集成路线
当前采用的是参考文档中“方式A：修改 SystemServer 源码，在 startOtherServices() 中添加自定义服务，并编译系统镜像”的路线。

已完成的关键接入包括：
1. 在 frameworks/base/services/core/java/com/android/server/permissionmonitor 下接入服务相关源码
2. 修改 SystemServer.java，加入 PermissionMonitorService 启动逻辑
3. 在 frameworks/base/core/res/AndroidManifest.xml 中增加自定义权限定义
4. 在 frameworks/base/core/res/res/values/strings.xml 中增加权限说明字符串
5. 调整 frameworks/base/services/core/Android.bp，使 permissionmonitor 并入 services.core 编译链路
6. 修复 AIDL 接入、Soong 依赖、SystemService 可见性等编译问题

四、当前关键技术判断
1. PermissionMonitorService 作为系统服务，更适合并入 services.core，而不是单独拆成独立 java_library 模块
2. PermissionRecord.aidl 属于 parcelable 声明型 AIDL，不适合和接口型 AIDL 一起按全量通配方式纳入 srcs
3. 当前 AOSP 分支下，服务的两阶段启动方式已经可以在 services.core 编译上下文中通过验证

五、关于 API 检查基线的结论
在完整系统整编过程中，新增权限定义触发了 framework public API 检查，出现了如下问题：
- 新增 android.Manifest.permission.ACCESS_PERMISSION_MONITOR 字段
- 与 frameworks/base/api/current.txt 不一致

结合参考文档与老师意见，当前结论如下：
1. 本实训目标强调框架开发、系统服务接入和平台源码修改
2. 当前采用的是修改 SystemServer 并编译完整系统镜像的方式，属于 framework 层集成路线
3. 因此，更新 API 检查基线是当前路线下可以接受的工程处理方式
4. 该操作应被理解为“平台 API 管理配套处理”，而不是本项目的核心功能点

六、后续建议执行顺序
1. 更新 API 检查基线
2. 继续低并发整编，避免因内存不足导致构建进程被系统 kill
3. 整编完成后，重新启动 emulator，确保加载最新镜像
4. 通过 logcat 和 service list 验证 PermissionMonitorService 是否在运行时成功启动并注册 Binder 服务

七、当前注意事项
1. 当前虚拟机环境整编时可能出现内存不足，建议优先使用较低并发，例如 m -j2
2. 运行时验证前应确保 emulator 使用的是最新构建产物，而不是旧实例残留镜像
3. 后续若用于答辩或文档总结，可将“更新 API 基线”解释为 framework 平台变更后的配套步骤
